# An 8bit MLX conversion emits doubled braces, and mlx-lm hides it

## Summary

`mlx-community/Qwen2.5-7B-Instruct-8bit` scores 7/50 on mlx-lm. The 4bit
conversion of the same model, on the same server, same corpus, same greedy
settings, scores 46/50. Nothing differs but the quantization.

The 8bit conversion emits malformed JSON in its tool calls. It doubles the
opening and closing braces:

```
8bit:  <tool_call>\n{{"name": "get_weather", "arguments": {"city": "Boston"}}}\n</tool_call>
4bit:  <tool_call>\n{"name": "get_weather", "arguments": {"city": "Boston"}}\n</tool_call>
```

That is not parseable JSON, and mlx-lm is right to reject it. This is a model
and conversion defect, not a server defect.

There is a separate, real server defect layered on top: when the parse fails,
mlx-lm returns HTTP 200 with `finish_reason: "tool_calls"` and an assistant
message containing neither `content` nor `tool_calls`. The client is told a tool
call happened, handed nothing, and given no error. The bytes are dropped, so
from the API alone the failure is indistinguishable from the model saying
nothing at all.

## How the bytes were recovered

They could not be recovered from the API, which is the point of the second
defect. They were recovered by loading the same weights directly through
`mlx_lm.load` / `mlx_lm.generate`, applying the same chat template with the same
tools that the harness sends, decoding greedily, and printing the raw string
before anything parses it. The 4bit run above is the control, generated by the
identical script with one string changed.

This step is not optional in this project. An earlier finding here claimed a
server was discarding valid tool calls; recovering the bytes showed the model
had emitted a malformed call and the server's parser was correct, and the claim
was retracted. The same procedure was applied before writing this, and it
changed the conclusion in the same direction: the parser is right, the model's
output is wrong.

## Evidence

- 5 runs per arm, greedy. `Qwen2.5-7B-Instruct-4bit` scored 46, 46, 46, 46, 46.
  `Qwen2.5-7B-Instruct-8bit` scored 7, 7, 7, 7, 7, with 30 scenarios classified
  `empty_response` on every run.
- mlx-lm logged `Failed to parse tool call (JSONDecodeError: Expecting property
  name enclosed in double quotes: line 1 column 2 (char 1))` 295 times across
  the 8bit runs, and 0 times across the 4bit runs. Column 2, character 1 is
  exactly where the second `{` sits.
- The server logs for the replication arms are committed at
  `evidence/mlxlm-server-qwen2.5-7b-8bit.log` (the 295 parse failures) and
  `evidence/mlxlm-server-qwen2.5-7b-4bit.log` (the zero-failure control).
  The only edit to the raw logs is a path redaction (`/Users/<user>` ->
  `/Users/REDACTED`) in one boilerplate warning line per file.
- `evidence/recover-mlx-raw-output.py` re-derives the raw generation that
  mlx-lm swallows: it applies the same chat template with the same tools and
  generates greedily with no parsing. Switching `REPO` to the 4bit conversion
  gives the control.
- The 7 scenarios the 8bit arm passes are the negative traps, where emitting no
  parsed call is the correct answer. That signature - full marks precisely where
  not calling a tool is right - is the same one that identified the granite
  case.

## Why this matters for the quantization question

The companion quantization case study found no monotonic relationship between
quantization level and tool-calling ability across GGUF quants of two Qwen2.5
models on llama.cpp. This result does not contradict that, and it is worth being
precise about why.

Quantization here did not gently degrade the model's ability to choose the right
tool. The 8bit model still knows exactly which function to call and with which
arguments - the recovered string is semantically perfect. What broke is the
literal output formatting, and it broke completely rather than gradually.

So the two findings say different things. Quantization level does not predict
tool-calling ability. A specific conversion artifact can still destroy a row
outright. The second is not a dose-response effect and cannot be extrapolated to
"8bit is worse than 4bit" - it is a property of this published conversion.

## Scope

- One model, one publisher's conversion (`mlx-community`), one server.
- Not a claim that 8bit MLX conversions are broken in general, or that this
  model is broken outside MLX. The GGUF Q8_0 of Qwen2.5-7B on llama.cpp scores
  47/50 and shows none of this.
- Not tested: whether re-converting the model from source with the current
  `mlx_lm.convert` reproduces the doubled braces, which would separate a bad
  published artifact from a converter bug. That is the obvious next step and it
  was not done here.

## Upstream status

Two separable items, neither filed:

1. The published `mlx-community/Qwen2.5-7B-Instruct-8bit` artifact emits
   malformed tool-call JSON. Belongs with whoever maintains that conversion.
2. mlx-lm reports `finish_reason: "tool_calls"` with an empty message and HTTP
   200 when its tool parser throws (`server.py`, the `ToolCallFormatter`
   `try/except` that logs a warning and continues). A client cannot detect this.
   This is the same defect class as the Ollama silent-discard issue this project
   reported previously, and it is what made byte recovery necessary here.

Recorded as candidate upstream issues. Filing gets its own duplicate-check pass
first.
